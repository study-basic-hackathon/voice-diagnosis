# CLAUDE.md — lover-app プロジェクト憲法

## プロジェクト概要

台本を読み上げてもらい、その「話し方」を音声解析して
相性のいい恋人像を生成するWebアプリ。

全員が同じ台本を読むことで話し方の個人差を公平に比較する。
同じ人が何度やっても似た結果になる「再現性」が最大の特徴。

---

## 技術スタック

### バックエンド
- Node.js + Express
- APIキーは .env で管理（フロントには露出させない）
- ポート：3000

### フロントエンド
- HTML / CSS / JavaScript（複数ファイルに分けてOK）
- Web Audio API（音声特徴の数値化）
- Web Speech API（テキスト変換）
- バックエンドとの通信：fetch API（REST）

## アーキテクチャ（処理フロー）
```
[1] 台本読み上げ
  画面に台本を表示。ユーザーが声に出して読む（約30秒〜1分）
  録音開始と同時に2系統で並行処理：
    - Web Audio API  → 話し方データをリアルタイム収集
    - Web Speech API → テキスト変換（読み飛ばし・詰まり検出用）

[2] 話し方データの数値化（ブラウザ側・Web Audio API）
  以下の指標をすべて数値として記録する：
  - 平均テンポ（mora/秒）：話す速さ
  - 無音区間の平均長さ（ms）：間の取り方
  - 無音区間の回数：どれだけ区切るか
  - 音量の標準偏差（dB）：抑揚の大きさ
  - 音量の平均値（dB）：声の大きさ
  - 音量変化の周期性：リズム感

[3] バックエンドに送信
  話し方データ6指標 ＋ 書き起こしテキスト を
  POST /api/analyze に送信

[4] 性格スコア算出（サーバー側・Claude API）
  受け取ったデータを Claude API に送信
  → ビッグファイブを 0〜100 でスコア化（JSON）

  プロンプトに含める対応関係のヒント：
  - テンポ速い・間少ない → 外向性高め
  - 抑揚大きい → 開放性高め
  - 音量安定・間が規則的 → 誠実性高め
  - 声が小さい・間が多い → 神経症傾向高め
  - 読み飛ばし少ない → 誠実性高め

[5] シード生成（再現性の核心）
  ビッグファイブの5スコアに重み付けして合計値を計算
  重み：開放性×0.3 + 誠実性×0.25 + 外向性×0.2 + 協調性×0.15 + 神経症傾向×0.1
  小数第1位で丸めた値をシードとして使用

[6] 相性計算（サーバー側）
  シードを使って相手の性格スコアを決定論的に算出
  - 協調性・誠実性 → 類似性重視（値が近いほど相性◎）
  - 外向性・開放性 → 相補性（差分を使う）
  - 神経症傾向 → 両者とも低い方が安定（上限60に補正）

[7] プロフィール生成（サーバー側・Claude API）
  相手の性格スコア + シード値 + temperature=0.3 で送信
  → 以下をJSONで返す：
    - name / age / job
    - appearance（見た目の特徴・3点）
    - personality（性格・3点）
    - catchphrase（口癖）
    - hobbies（趣味・2〜3個）
    - meet_story（出会いエピソード）
    - episodes（あるあるエピソード・3つ）
```

## 台本（全ユーザー共通・変更不可）

「先日、久しぶりに友人と会いました。
特に予定はなかったのですが、気づいたら夕方まで話し込んでいました。
帰り道、少し遠回りして歩いて帰ったのですが、
その時間がなんだか心地よかったです。
最近、こういう何でもない時間が一番好きだなと思います。
あなたはどんな時間が好きですか？」

## 画面構成（4画面）

### 画面1：TOP
- タイトルとコンセプト説明
- 「はじめる」ボタン

### 画面2：録音
- 台本を大きく表示
- 「録音開始」ボタン
- 録音中：リアルタイム音量ビジュアライザー
- 「録音停止」ボタン

### 画面3：分析中
- ローディングアニメーション
- 進捗メッセージ

### 画面4：結果
- 自分のビッグファイブレーダーチャート
- 恋人プロフィールカード
- 出会いエピソード
- あるあるエピソード3つ
- 「もう一度やる」ボタン

## UI/UXの方針

- スマホ・PC両対応（レスポンシブ）
- 配色：ロマンティックで落ち着いたトーン
  （ベース：ディープネイビーorソフトベージュ、アクセント：ローズピンク）
- エラーハンドリング必須：
  - マイク許可拒否
  - ブラウザ非対応
  - API失敗
  - 録音が短すぎる（10秒未満は警告）

---

## コード品質基準

- コメントは日本語で書く
- APIキーは絶対にフロントに書かない（.envで管理）
- Claude API呼び出しはasync/awaitで統一
- モックモード必須：
  .envに MOCK_MODE=true を設定するとダミーデータで
  全画面を通しでテストできること
